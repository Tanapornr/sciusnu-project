# โครงสร้างข้อมูลสำหรับหน้าอาจารย์ที่ปรึกษา

เอกสารนี้แยกข้อมูลหลักของระบบออกเป็น 4 หมวด เพื่อให้หน้าอาจารย์ที่ปรึกษาเรียกใช้ข้อมูลได้ชัดเจน และอธิบายฐานข้อมูลได้ง่ายในการนำเสนอ

## 1. ข้อมูลโครงงาน

เก็บที่ชีต `NEW_Projects`

| ตัวแปร | ความหมาย |
| --- | --- |
| `projectId` | รหัสโครงงาน ใช้เป็น primary key |
| `title` | ชื่อโครงงาน |
| `studentIds` | รหัสนักเรียนในกลุ่ม คั่นด้วย `,` |
| `studentNames` | ชื่อนักเรียนในกลุ่ม คั่นด้วย `,` |
| `advisorId` | อีเมลหรือบัญชีอาจารย์ที่ปรึกษาหลัก |
| `coAdvisorId` | อีเมลหรือบัญชีอาจารย์ที่ปรึกษาร่วม |
| `schoolAdvisorId` | อีเมลหรือบัญชีอาจารย์ที่ปรึกษาโรงเรียน |
| `school` | สาขา/โรงเรียน/กลุ่มโครงงาน |
| `dueDate` | กำหนดส่งหลัก |
| `status` | สถานะรวมของโครงงาน |

ตัวเลือก `status`: `กำลังดำเนินการ`, `ส่งงานแล้ว`, `รอตรวจ`, `อนุมัติ`, `ขอแก้ไข`, `เสร็จสิ้น`

## 2. ข้อมูลนักเรียน

เก็บบัญชีเข้าสู่ระบบที่ `NEW_Users` และเก็บโปรไฟล์นักเรียนที่ `NEW_Students`

| ตัวแปร | ความหมาย |
| --- | --- |
| `studentId` | รหัสนักเรียน ใช้เป็น key หลักของนักเรียน |
| `userId` | user สำหรับ login ส่วนมากเท่ากับรหัสนักเรียน |
| `prefix` | คำนำหน้าชื่อ |
| `name` | ชื่อ-นามสกุล |
| `email` | อีเมลนักเรียน |
| `phone` | เบอร์โทร |
| `school` | โรงเรียน/สาขา |
| `classLevel` | ระดับชั้น |
| `room` | ห้อง |
| `photoUrl` | รูปโปรไฟล์ |
| `projectIds` | โครงงานที่นักเรียนอยู่ |
| `active` | เปิด/ปิดการใช้งาน |

เหตุผลที่แยก `NEW_Users` กับ `NEW_Students`: `NEW_Users` ใช้ยืนยันตัวตนและรหัสผ่าน ส่วน `NEW_Students` ใช้แสดงโปรไฟล์ในระบบ

## 3. ข้อมูลผู้เชี่ยวชาญ

เก็บผู้เชี่ยวชาญที่ `NEW_Experts` และความสัมพันธ์กับโครงงานที่ `NEW_ProjectExperts`

### `NEW_Experts`

| ตัวแปร | ความหมาย |
| --- | --- |
| `expertId` | รหัสผู้เชี่ยวชาญ หรืออีเมล |
| `name` | ชื่อ-นามสกุล |
| `email` | อีเมล |
| `phone` | เบอร์โทร |
| `organization` | หน่วยงาน |
| `expertise` | ความเชี่ยวชาญ เช่น ชีววิทยา วิศวกรรม สถิติ |
| `expertRole` | บทบาทผู้เชี่ยวชาญ |
| `photoUrl` | รูปโปรไฟล์ |
| `active` | เปิด/ปิดการใช้งาน |

### `NEW_ProjectExperts`

| ตัวแปร | ความหมาย |
| --- | --- |
| `projectExpertId` | key ของความสัมพันธ์ |
| `projectId` | โครงงานที่เกี่ยวข้อง |
| `expertId` | ผู้เชี่ยวชาญที่เกี่ยวข้อง |
| `expertRole` | บทบาทในโครงงาน |
| `responsibility` | หน้าที่รับผิดชอบ |
| `assignedAt` | วันที่มอบหมาย |
| `active` | ยังเกี่ยวข้องกับโครงงานหรือไม่ |

ตัวเลือก `expertRole`: `advisor`, `co_advisor`, `school_advisor`, `external_expert`, `content_expert`, `methodology_expert`

## 4. ข้อมูลกระบวนการ Progress

เก็บขั้นตอนมาตรฐานที่ `NEW_ProgressSteps` และสถานะแต่ละโครงงานที่ `NEW_ProjectProgress`

### `NEW_ProgressSteps`

| ตัวแปร | ความหมาย |
| --- | --- |
| `processId` | รหัสขั้นตอน |
| `stepOrder` | ลำดับขั้นตอน |
| `title` | ชื่อขั้นตอน |
| `description` | คำอธิบาย |
| `workType` | ประเภทงานที่เชื่อมกับการส่งงาน |
| `requiredFileType` | ประเภทไฟล์ที่ต้องส่ง เช่น `pdf` |
| `defaultStatus` | สถานะเริ่มต้น |
| `sortOrder` | ลำดับแสดงผล |
| `active` | เปิด/ปิดขั้นตอน |

ขั้นตอนเริ่มต้น:

1. `proposal` — โครงร่าง (Proposal)
2. `progress_1` — รายงานความก้าวหน้า
3. `chapter_1_3` — บทที่ 1-3
4. `final_report` — รายงานฉบับสมบูรณ์
5. `presentation` — สไลด์นำเสนอ

### `NEW_ProjectProgress`

| ตัวแปร | ความหมาย |
| --- | --- |
| `projectProgressId` | key รูปแบบ `projectId-processId` |
| `projectId` | รหัสโครงงาน |
| `processId` | ขั้นตอนที่อ้างอิง |
| `status` | สถานะของขั้นตอน |
| `percent` | เปอร์เซ็นต์ความคืบหน้า |
| `dueDate` | กำหนดส่งของขั้นตอน |
| `startedAt` | วันที่เริ่ม |
| `submittedAt` | วันที่ส่งงาน |
| `reviewedAt` | วันที่ตรวจ |
| `completedAt` | วันที่เสร็จสิ้น |
| `ownerId` | ผู้รับผิดชอบ/ผู้ส่ง |
| `reviewerId` | ผู้ตรวจ |
| `submissionId` | รายการส่งงานที่เชื่อมอยู่ |
| `note` | หมายเหตุ |

ตัวเลือก `status`: `not_started`, `in_progress`, `submitted`, `reviewing`, `revision`, `approved`, `completed`

## ทางเลือกฐานข้อมูล

| ทางเลือก | เหมาะกับ | ข้อดี | ข้อจำกัด |
| --- | --- | --- | --- |
| Google Sheets | โครงงานระดับโรงเรียน/ต้นแบบจริง | แก้ไขง่าย เห็นข้อมูลทันที ใช้กับ Apps Script ได้เร็ว | สิทธิ์และข้อมูลจำนวนมากต้องจัดการระวัง |
| Google Sheets + Apps Script | ระบบที่ใช้ในงานจริงขนาดเล็ก-กลาง | มี backend, อีเมล, Drive, PDF ในชุดเดียว | ไม่เหมาะกับ transaction ซับซ้อนมาก |
| Firebase/Supabase | ระบบที่มีผู้ใช้เยอะและต้อง real-time | auth และฐานข้อมูลแข็งแรงกว่า | ต้องตั้งค่าเพิ่มและซับซ้อนกว่า |
| SQL Database | ระบบองค์กร/ต่อยอดระยะยาว | โครงสร้างชัด สืบค้นเร็ว ทำรายงานได้ดี | ต้องมี server/API และคนดูแลฐานข้อมูล |

แนวทางที่ใช้ตอนนี้คือ `Google Sheets + Apps Script` เพราะเหมาะกับการนำเสนอและใช้งานจริงเร็วที่สุด โดยยังแยกตารางแบบ normalized เพื่อให้ย้ายไปฐานข้อมูลอื่นได้ในอนาคต
